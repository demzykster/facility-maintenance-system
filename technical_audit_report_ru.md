# Технический аудит системы обслуживания, техники и уборки

Дата: 16.06.2026  
Статус проекта: демо / прототип на React, без production-backend

## 1. Короткий вывод

Система выглядит сильной как демонстрация автоматизированного CMMS-процесса: есть роли, заявки, SLA, техника, документы, уборка, история действий, подтверждения, финальное закрытие и базовая аналитика.

Главная проблема: сейчас это клиентское приложение, где данные, роли, пароли, бизнес-логика и права находятся на стороне браузера. Для теста это нормально. Для реальной эксплуатации так запускать нельзя.

Если говорить честно: логика процесса хорошая, но архитектурное ядро надо вынести на сервер. Иначе любой пользователь с доступом к браузеру потенциально может изменить данные, роли, статусы, стоимость закрытия и историю.

## 2. Что уже хорошо

### Процесс заявок

- Есть понятная цепочка: работник создает заявку -> руководитель/админ проверяет -> техник выполняет -> руководитель подтверждает -> админ финально закрывает.
- Есть история действий по заявке.
- Есть SLA и визуальное выделение просрочек.
- Есть отдельная логика для техники/погрузчиков: простой, замена, критичность, документы, обслуживание.
- Есть ветка “техник не получил инструмент/технику” -> руководитель видит, что требуется передача.
- Есть отрицательная ветка: руководитель может вернуть заявку технику с обязательной причиной.
- Есть финальное закрытие админом с качеством закрытия и стоимостью.

### Роли

- Админ видит общую картину, критичные заявки, документы, аналитику.
- Руководитель видит задачи своего уровня: подтверждение, заявки отдела, состояние уборки.
- Техник видит свои работы и задачи к принятию.
- Работник видит простую форму создания и свои отчеты.
- Работник уборки видит текущие обходы и свои зоны.

### Уборка

- Появилась отдельная логика зон уборки.
- У зоны есть расписание обходов.
- Работник уборки видит “что делать сейчас”.
- В обходе есть чек-лист, фото и заметка.
- Админ видит выполнение по дням, пропущенные обходы и открытые отчеты.

### Данные и отчеты

- Есть импорт/экспорт резервной копии.
- Есть Excel/CSV/печать.
- Есть демо-данные для показа.
- Есть учет документов по технике.

## 3. Критичные проблемы перед реальным запуском

### 3.1. Нет настоящего backend

Сейчас основное хранение работает через `window.storage` / `localStorage`, а fallback держит данные в памяти браузера.

Риск:

- данные можно изменить прямо из браузера;
- нет серверной проверки прав;
- нет надежной синхронизации между пользователями;
- нет нормальной истории изменений;
- нет транзакций;
- при очистке браузера данные могут пропасть;
- несколько пользователей могут перезаписать друг друга.

Как должно быть:

- отдельный backend API;
- база данных;
- серверная авторизация;
- серверная проверка всех переходов статусов;
- журнал аудита, который нельзя редактировать из клиента;
- хранение файлов отдельно от основной базы.

### 3.2. Пароли и PIN находятся на клиенте

В коде есть демо-пользователи и пароли вроде `1234`. Логика входа работает на клиенте.

Риск:

- пользователь может посмотреть или изменить учетные данные;
- можно подменить роль;
- можно войти как админ через манипуляции с локальными данными;
- нет реальной безопасности.

Как должно быть:

- пароли только на сервере, только в виде хэшей;
- PIN тоже хэшировать;
- сессии через защищенные токены;
- роли и права выдаются сервером;
- все действия проверяются сервером, а не интерфейсом.

### 3.3. Role switch админа опасен для production

В демо это удобно: админ может переключаться между ролями и тестировать процесс.

Риск в реальной системе:

- если такой механизм останется в production, можно ошибочно выполнить действие от имени другой роли;
- аудит станет неясным: кто реально нажал кнопку.

Как должно быть:

- в production отключить;
- если нужен режим “действовать от имени”, он должен быть отдельным admin-инструментом;
- каждое действие должно писать: “реальный пользователь: админ, действовал как: техник/работник”.

### 3.4. Импорт backup без строгой валидации

Система умеет импортировать JSON backup.

Риск:

- можно импортировать поврежденные данные;
- можно подменить пользователей, роли, статусы, стоимость, историю;
- можно загрузить слишком большой файл и повесить браузер;
- можно внести некорректные связи: заявка на несуществующую технику, зона без уборщика и т.д.

Как должно быть:

- импорт только для админа;
- схема валидации данных;
- лимит размера файла;
- dry-run перед импортом;
- лог импорта;
- резервная копия перед восстановлением;
- запрет импорта пользователей/ролей без отдельного подтверждения.

### 3.5. Фото хранятся base64 в клиентском хранилище

Фото уменьшаются и сохраняются как base64.

Риск:

- быстро раздувается localStorage/storage;
- браузер может начать тормозить;
- backup становится тяжелым;
- потеря данных при превышении лимита хранилища;
- нет нормальной политики доступа к фото.

Как должно быть:

- файлы хранить в object storage: S3, Azure Blob, серверное файловое хранилище;
- в базе хранить только ссылку, размер, mime-type, checksum;
- ограничить размер, типы файлов и количество;
- делать thumbnail отдельно.

### 3.6. Нет серверной целостности переходов статусов

В коде есть хорошая таблица переходов `TRANSITIONS`, но фактические изменения статусов выполняются на клиенте.

Риск:

- можно вручную поставить любой статус;
- можно закрыть заявку без нужного подтверждения;
- можно удалить или изменить историю;
- можно задним числом изменить стоимость.

Как должно быть:

- workflow engine на сервере;
- каждый переход статуса через API;
- сервер проверяет: роль, текущий статус, обязательные поля, владельца действия;
- audit log append-only.

## 4. Найденные функциональные проблемы

### 4.1. Модуль уборки: неправильный текст на иврите

На экране уборки используется слово `זונות` вместо корректного `אזורים` / `אזורי ניקיון`.

Это критично для презентации и production. Исправить срочно во всех местах:

- вкладка зон;
- пустые состояния;
- “мои зоны”;
- зоны отдела;
- подсказки в настройках пользователей.

### 4.2. Уборку можно завершить с чек-листом 0/6

Я открыл обход уборки и нажал завершение при `0/6`. Система приняла обход как выполненный.

Риск:

- уборщик может закрыть обход без фактической проверки;
- начальство увидит “выполнено”, но качество не подтверждено.

Рекомендация:

- либо блокировать завершение, пока не отмечены все обязательные пункты;
- либо ввести режим “частично выполнено” с обязательной причиной;
- либо позволять 0/6 только администратору/ответственному с пометкой “закрыто без чек-листа”.

### 4.3. Возврат заявки по зданию может визуально зависнуть

После возврата `F-905` руководителем заявка получила статус “в работе”, но визуально оказалась не там, где ожидается. У транспортного техника она не появилась, потому что это заявка по зданию.

Риск:

- заявка формально в работе, но исполнитель может ее не увидеть;
- руководитель думает, что вернул технику, а фактически непонятно кому.

Рекомендация:

- разделить исполнителей: техник транспорта, техник здания, подрядчик, админ;
- при возврате направлять конкретному исполнителю;
- если исполнителя нет, переводить в очередь “требуется назначение”.

### 4.4. Ветка “техник не получил инструмент” требует явного действия руководителя

Система правильно показывает руководителю, что инструмент/техника не переданы. Но на стороне руководителя не хватает явной кнопки действия.

Рекомендация:

- добавить кнопку “инструмент передан технику”;
- либо статус “ожидаем подтверждения техника”;
- в истории писать два события: “руководитель передал” и “техник подтвердил получение”.

### 4.5. Фото обязательно для работника

Обычный работник не может отправить заявку без фото.

Плюс:

- повышает качество диагностики.

Минус:

- если камера не работает, плохая связь, темное место, сотрудник не сможет создать заявку.

Рекомендация:

- оставить фото обязательным для критичных/визуальных дефектов;
- добавить вариант “нет возможности приложить фото” с обязательным объяснением;
- руководитель потом решает, принять или вернуть на уточнение.

## 5. Архитектурные проблемы

### 5.1. Один большой React-файл

Сейчас почти вся система находится в `src/ClaudeMaintenanceApp.jsx`: данные, UI, роли, workflow, отчеты, уборка, техника, импорт, настройки.

Риск:

- сложно поддерживать;
- легко сломать одно место изменением другого;
- сложно покрыть тестами;
- сложно подключать backend;
- новые разработчики будут долго входить в проект.

Как лучше:

- разделить на модули:
  - `domain/tickets`;
  - `domain/fleet`;
  - `domain/cleaning`;
  - `domain/users`;
  - `services/api`;
  - `services/storage`;
  - `components`;
  - `pages`;
  - `workflow`;
  - `reports`.

### 5.2. Бизнес-логика смешана с интерфейсом

Сейчас функции изменения статусов, расчета SLA, риска и прав находятся рядом с компонентами.

Риск:

- нельзя надежно тестировать правила отдельно;
- сложно перенести правила на сервер;
- UI может случайно обойти правило.

Как лучше:

- вынести правила в отдельное ядро:
  - `canTransition(from, to, actor, ticket)`;
  - `calculateSla(ticket, config)`;
  - `calculateRisk(ticket)`;
  - `getAvailableActions(actor, ticket)`;
  - `validateTicketAction(action)`.

### 5.3. Нет типов и схем

Проект на JSX без TypeScript и без runtime-схем данных.

Риск:

- легко записать объект с неправильной структурой;
- импорт может сломать интерфейс;
- ошибка в поле статуса проявится только в браузере.

Как лучше:

- TypeScript;
- Zod/Yup/JSON Schema для API и импорта;
- единые enum для ролей, статусов, типов заявок.

### 5.4. Нет автоматических тестов

Я не увидел тестового слоя.

Нужно добавить:

- unit-тесты правил workflow;
- тесты SLA и просрочек;
- тесты ролей и прав;
- e2e-тесты основных сценариев;
- тест импорта поврежденного backup;
- тест больших списков техники и заявок.

## 6. Безопасность

### Срочные риски

- Авторизация только на клиенте.
- Пароли/PIN в открытом виде.
- Роли можно потенциально подменить в локальном хранилище.
- Действия не проверяются сервером.
- История заявки редактируема через данные.
- Backup может содержать чувствительные данные и фото.
- Внешние ссылки на документы открываются без серверного контроля.
- AI-интеграция, если будет включена, может отправлять внутренние данные наружу.

### Что должно быть в production

- HTTPS.
- Backend auth.
- RBAC/ABAC права.
- Серверная проверка каждого действия.
- Audit log append-only.
- Хэширование паролей.
- Rate limiting.
- CSRF/XSS защита.
- CSP headers.
- Логи входов и неудачных попыток.
- Разделение demo/prod.
- Шифрование backup.
- Ограничение доступа к фото и документам.

## 7. База данных: как должно быть

Сейчас база тестовая/локальная. Для production нужна нормальная реляционная база, например PostgreSQL.

Минимальные таблицы:

- `users`
- `roles`
- `departments`
- `tickets`
- `ticket_events`
- `ticket_photos`
- `assets`
- `asset_documents`
- `preventive_maintenance`
- `inspections`
- `cleaning_zones`
- `cleaning_rounds`
- `cleaning_complaints`
- `notifications`
- `attachments`
- `audit_log`
- `settings`

Ключевые принципы:

- `ticket_events` нельзя редактировать, только добавлять.
- В `tickets` хранится текущий статус, но вся история в `ticket_events`.
- Фото и файлы не хранить base64 в основной таблице.
- Для документов техники хранить срок действия, тип, ссылку на файл, кто загрузил, когда проверено.
- Для уборки хранить каждый обход как отдельную запись.
- Для критичных операций использовать транзакции.

Индексы:

- `tickets(status, due_at)`
- `tickets(track, status)`
- `tickets(asset_id)`
- `tickets(created_by)`
- `ticket_events(ticket_id, created_at)`
- `asset_documents(asset_id, expires_at)`
- `cleaning_rounds(zone_id, at)`
- `cleaning_complaints(status, zone_id)`

## 8. Нагрузка и производительность

### Сейчас

Приложение каждые 15 секунд перечитывает коллекции из storage и сравнивает данные через `JSON.stringify`.

Для демо нормально.

При росте данных будет проблема:

- 1 000 заявок еще терпимо;
- 10 000 заявок начнут тормозить;
- фото в base64 быстро убьют производительность;
- Excel/печать больших отчетов в браузере будет зависать;
- аналитика на клиенте станет тяжелой.

### Как должно быть

- Пагинация на сервере.
- Фильтры на сервере.
- Индексы в базе.
- Отдельный API для аналитики.
- Кэширование справочников.
- WebSocket/SSE для уведомлений вместо постоянной перечитки всего.
- Фоновые задачи для SLA, документов, напоминаний.
- Генерация тяжелых отчетов на сервере.

## 10. QA-рекомендации

### Обязательные тест-сценарии

1. Работник создает заявку без фото / с фото / с ошибкой.
2. Руководитель принимает заявку.
3. Руководитель возвращает заявку работнику.
4. Руководитель отклоняет заявку.
5. Техник принимает работу.
6. Техник не получил инструмент.
7. Руководитель передал инструмент.
8. Техник завершает.
9. Руководитель подтверждает.
10. Руководитель возвращает технику.
11. Админ финально закрывает со стоимостью.
12. Заявка уходит в SLA-просрочку.
13. Документ техники истек.
14. Уборщик закрывает обход полностью.
15. Уборщик закрывает обход частично.
16. Уборщик пытается закрыть обход 0/6.
17. Анонимный QR-отчет создает complaint.
18. Админ подтверждает complaint и превращает в заявку.
19. Backup export/import.
20. Поврежденный backup не ломает систему.

### Негативные тесты

- пользователь без прав пытается открыть админский экран;
- техник пытается закрыть заявку финально;
- работник пытается изменить чужую заявку;
- импорт с неправильным статусом;
- огромная фотография;
- 1000+ заявок;
- сеть пропала во время сохранения;
- два пользователя меняют одну заявку одновременно.

## 12. Финальная оценка

Как демо для начальства: сильная база, хороший визуальный эффект, понятная автоматизация процессов.

Как production-система: пока рано. Нужно построить серверное ядро, базу, права, аудит и надежное хранение файлов.

Главная ценность проекта уже видна: он правильно показывает, кто должен действовать на каждом этапе. Главная опасность тоже ясна: сейчас браузер слишком “всемогущий”. Для настоящей системы браузер должен только показывать и отправлять команды, а решать и сохранять должен сервер.
