# Дополнение к аудиту QA/QC после новой версии

Дата анализа: 16.06.2026  
Файлы: `maintenance-app (2).jsx`, `CLAUDE (1).md`

## 1. Короткий вывод

Новая версия стала заметно сильнее именно по процессам: модуль уборки теперь ближе к реальной системе, а не просто к демо. Появились зоны, дни активности, окна уборки, чек-листы по окнам, жалобы, анонимный QR-репорт, маршрутизация ответственности, эскалация, статусы “в работе”, замещение и отсутствие сотрудника.

Но из-за этого система перешла на следующий уровень риска: теперь важно тестировать не только “кнопка работает / экран открывается”, а бизнес-правила. Главный QA/QC-фокус: кто владелец события, кто имеет право закрыть, когда создается заявка на ремонт, как считаются SLA/просрочки, и нет ли состояний, где объект висит без ответственного.

## 2. Что уже стало лучше

1. Терминология уборочных зон исправлена.
   - Старый критический риск с ивритским словом `זונות` устранен.
   - Теперь используются корректные варианты `אזור`, `אזורים`, `אזורי ניקיון`.
   - Это нужно закрепить регрессионным тестом, чтобы ошибка не вернулась при новых правках.

2. Нельзя закрыть уборочный обход с пустым чек-листом.
   - В `RoundForm` добавлена логика: каждый пункт должен быть либо отмечен как выполненный, либо отмечен как проблема с причиной.
   - Это исправляет прошлую дыру, где можно было отправить 0/6 и система считала обход завершенным.

3. Появилась нормальная логика жалоб.
   - Менеджер/админ открывают жалобу сразу.
   - Работник/анонимный пользователь/уборщик идут через модерацию.
   - Жалоба от уборщика становится задачей админа, а не возвращается самому уборщику.
   - Поломка после подтверждения создает facility-заявку.

4. У менеджера появилась видимость уборки.
   - Видны зоны, окна, статус по времени, открытые жалобы, спецификация зоны и возможность сообщить о проблеме.

5. Добавлены дни активности и окна уборки.
   - Это правильно для реальной эксплуатации: не все зоны должны убираться каждый день.

6. Добавлена логика отсутствия и замещения.
   - Это важная функция для реального объекта, потому что уборка часто ломается не из-за задачи, а из-за смен, отпусков и подмен.

## 3. Что остается критичным для production

1. Данные, роли и бизнес-правила все еще живут на клиенте.
   - Используется `window.storage` с fallback в память.
   - Для демо это нормально.
   - Для production это нельзя считать настоящей системой учета.
   - Нужны backend, база данных, серверная авторизация, серверные проверки переходов статусов и неизменяемый журнал событий.

2. В коде остаются встроенные demo-логины и простые пароли/PIN.
   - Например `1234`, встроенный админ, менеджер, техник, работник, уборщик.
   - Для демонстрации допустимо.
   - Для пилота на реальных данных должно быть отключено через production mode.

3. Функция “запомнить меня” сохраняет пароль/PIN.
   - Сейчас сохраняются email/password/code/workerNo в локальном хранилище.
   - В production нельзя хранить пароль или PIN на устройстве в открытом виде.
   - Максимум: помнить логин, но не секрет.

4. Есть прямой вызов Anthropic API из клиентского кода.
   - Если это останется в production, API-ключ и данные могут уйти на сторону клиента.
   - AI-вызовы должны идти через backend-proxy с логированием, лимитами, очисткой данных и контролем доступа.

5. Export/print через XLSX/HTML требует защиты.
   - Нужно проверять Excel formula injection: значения, начинающиеся с `=`, `+`, `-`, `@`.
   - Нужно проверять HTML/print-output, чтобы пользовательский текст не превращался в исполняемый HTML.

6. Backup/import должен иметь строгую валидацию схемы.
   - Сейчас импорт резервной копии потенциально может сломать структуру данных.
   - Нужен dry-run, проверка версии схемы, список изменений перед импортом и rollback.

## 4. Новые QA/QC-риски после изменений

### 4.1 Чек-лист уборки: “выполнено” и “проблема” одновременно

Сейчас пункт чек-листа может быть одновременно отмечен как выполненный и как проблема.

Нужно принять бизнес-решение:

1. Вариант A: это запрещено.
   - Тогда при выборе “проблема” галочка “выполнено” должна сниматься.
   - При выборе “выполнено” проблема должна очищаться.

2. Вариант B: это разрешено.
   - Тогда UI должен ясно объяснять смысл: “выполнено, но есть дефект”.
   - В отчетах такие пункты нельзя считать просто чисто выполненными.

Моя рекомендация: для руководства и контроля качества лучше запретить двойной статус. Один пункт = одно состояние: выполнено / проблема-грязь / проблема-поломка.

### 4.2 Жалоба и ремонтная заявка могут жить разными жизнями

Если жалоба типа “поломка” подтверждена, создается facility-заявка. Но надо четко определить связь:

1. Закрытие жалобы закрывает ли ремонтную заявку? Скорее нет.
2. Закрытие ремонтной заявки закрывает ли жалобу? Желательно да, но с отдельной записью.
3. Кто видит статус связанной заявки?
4. Что делать, если заявка на ремонт отклонена?
5. Что делать, если жалобу отклонили, но ticket уже создан?

Нужно добавить правило: broken complaint и facility ticket связаны через отдельную связь, а финальное закрытие должно синхронизироваться или хотя бы отображаться в обе стороны.

### 4.3 Статусы жалоб стали сложными

Сейчас есть несколько измерений:

- `status`: pending / open / resolved / rejected
- `ownerRole`: cleaner / admin
- `escalatedTo`: admin
- `progress`: in_progress
- `ticketId`: есть/нет связанной заявки

Это правильно, но требует матрицы состояний. Без нее появятся зависшие ситуации.

Минимальное QC-правило:

- У каждой открытой жалобы должен быть один ответственный владелец.
- Жалоба не должна быть одновременно “у уборщика” и “у админа”.
- Эскалация должна менять видимость и действия, а не просто добавлять флаг.
- Для каждого статуса должен быть следующий допустимый шаг.

### 4.4 Окна уборки требуют boundary-тестов

Логика `zoneTodayStatuses` делит день на окна и статусы:

- pending
- due
- overdue
- missed
- done

Нужно тестировать границы времени:

1. За минуту до начала окна.
2. Ровно в начале окна.
3. Ровно в плановое время.
4. Ровно на границе tolerance.
5. Через минуту после tolerance.
6. Перед следующим окном.
7. После конца слота.
8. Несколько окон в один день.
9. Обход без `winId`, но попавший в интервал по времени.
10. Обход после полуночи.

### 4.5 Дни активности могут изменить старые зоны

В логике есть важный нюанс:

- старые зоны без `activeDays` считаются активными все 7 дней;
- но при открытии формы редактирования новая default-логика может поставить рабочую неделю `א׳–ה׳`.

Риск: старую зону открыли, сохранили без намерения менять расписание, и она стала работать только 5 дней вместо 7.

Нужно проверить и, если подтвердится, исправить миграцией или явным UI-предупреждением.

### 4.6 Анонимный QR-репорт пока защищен только локально

Сейчас есть throttle 45 секунд на устройстве. Это слабая защита:

- можно открыть incognito;
- можно очистить storage;
- можно отправлять с другого устройства;
- можно автоматизировать отправку.

Для production нужно:

- server-side rate limit;
- QR-token на конкретную зону;
- ограничение размера фото;
- антиспам;
- возможно CAPTCHA или внутренний Wi-Fi allowlist;
- журнал отклоненных жалоб.

### 4.7 Отсутствие и замещение

Функция важная, но требует правил:

1. Если уборщик отсутствует, кто получает уведомление?
2. Просрочка считается на отсутствующего или на ответственного за подмену?
3. Можно ли закрыть чужую зону без явного назначения?
4. Как отображается “cover” в аналитике?
5. Что делать с отсутствием, добавленным задним числом?
6. Что делать, если отсутствие удалили после пропуска?

Рекомендация: хранить отдельное событие “замещение”, а не только флаг в обходе.

## 5. Что добавить в QA-тестирование

### 5.1 Минимальный regression pack

1. Логин каждой ролью:
   - admin
   - manager
   - tech
   - worker
   - cleaner
   - anonymous report

2. Facility-заявка:
   - создать;
   - назначить;
   - взять в работу;
   - поставить ожидание;
   - вернуть;
   - закрыть с фото;
   - проверить лог и SLA.

3. Transport-заявка:
   - создать по технике;
   - проверить, что она не смешивается с building/facility-процессом;
   - проверить “vehicle not handed over”;
   - проверить счетчик ожидания техники.

4. Уборка:
   - создать зону;
   - назначить уборщика;
   - настроить дни и окна;
   - пройти обход полностью;
   - пройти обход с грязью;
   - пройти обход с поломкой;
   - проверить создание жалобы;
   - проверить создание facility-ticket после approve.

5. Жалобы:
   - manager -> open -> cleaner resolve;
   - worker -> pending -> admin approve -> cleaner resolve;
   - anonymous -> pending -> reject;
   - cleaner dirty issue -> pending -> admin owned -> admin resolve;
   - cleaner broken issue -> pending -> approve -> facility ticket.

6. Замещение:
   - cleaner A отсутствует;
   - cleaner B делает cover;
   - аналитика показывает cover отдельно;
   - уведомления не обвиняют отсутствующего сотрудника без пометки.

7. Экспорт:
   - Excel export;
   - CSV fallback;
   - печать;
   - текст с символами `= + - @ < >`;
   - длинные строки;
   - иврит/RTL.

8. Backup/import:
   - валидный backup;
   - backup старой версии;
   - поврежденный JSON;
   - отсутствующие поля;
   - дубликаты ID;
   - импорт без потери существующих данных.

### 5.2 QC-проверки перед показом руководству

1. Ни один процесс не должен иметь “мертвую ветку”.
2. На каждом экране должно быть понятно:
   - кто ответственный;
   - что следующий шаг;
   - кто может закрыть;
   - что будет после закрытия.
3. Любая просрочка должна иметь причину:
   - исполнитель;
   - менеджер;
   - админ;
   - ожидание внешнего фактора;
   - отсутствие/замещение.
4. Любое закрытие должно иметь след:
   - кто;
   - когда;
   - действие;
   - фото/причина, если требуется.
5. Любая автоматическая заявка должна иметь ссылку на источник:
   - из какой жалобы;
   - из какого обхода;
   - из какой зоны.

## 6. Что нужно изменить в архитектуре для настоящей системы

### 6.1 Backend

Нужен серверный слой, который отвечает за:

- авторизацию;
- роли и права;
- переходы статусов;
- SLA;
- создание связанных заявок;
- audit log;
- rate limit;
- импорт/экспорт;
- интеграцию с почтой/WhatsApp/Teams/SMS;
- хранение файлов.

Клиент не должен сам решать, можно ли закрыть заявку, кому она принадлежит и какой статус следующий.

### 6.2 База данных

Рекомендуемые сущности:

- users
- roles
- departments
- tickets
- ticket_events
- ticket_status_history
- ticket_waiting_periods
- ticket_attachments
- fleet_units
- fleet_documents
- preventive_maintenance_tasks
- inspections
- cleaning_zones
- cleaning_zone_windows
- cleaning_checklist_items
- cleaning_rounds
- cleaning_round_items
- cleaning_round_issues
- cleaning_complaints
- complaint_events
- complaint_ticket_links
- cleaning_absences
- cleaning_cover_assignments
- qr_tokens
- notification_events
- import_jobs
- export_jobs
- audit_log

Важно: для отчетности лучше не хранить все только JSON-объектами. JSON можно оставить для snapshot, но ключевые события и строки чек-листа лучше нормализовать.

### 6.3 Audit log

Нужен append-only журнал:

- пользователь;
- роль;
- действие;
- старое значение;
- новое значение;
- IP/device;
- время;
- источник действия: UI/API/import/system.

Это важно не только для безопасности, но и для споров: “кто закрыл”, “почему просрочено”, “когда менеджер получил уведомление”.

## 7. Приоритеты исправления

### Срочно перед реальным пилотом

1. Убрать production-доступ через встроенные demo-логины.
2. Не сохранять пароль/PIN в `remember me`.
3. Перенести роли, права и переходы статусов на backend.
4. Сделать серверный audit log.
5. Закрыть матрицу жалоб: status + ownerRole + escalation + ticket.
6. Решить двойной статус чек-листа: выполнено и проблема одновременно.
7. Добавить валидацию backup/import.
8. Добавить защиту export от Excel formula injection.

### Желательно доработать до презентации руководству

1. Подготовить дерево процессов для:
   - facility/building;
   - transport;
   - cleaning round;
   - cleaning complaint;
   - anonymous QR report;
   - absence/cover.
2. Добавить видимый “след решения” в карточках:
   - кто сейчас владелец;
   - почему задача ожидает;
   - сколько времени на исполнителе/менеджере/админе.
3. Сделать тестовый сценарий “один день работы системы”:
   - заявка;
   - уборка;
   - жалоба;
   - поломка;
   - ожидание;
   - закрытие;
   - отчет.

### Для масштабирования

1. Индексы в БД по:
   - status;
   - assignee;
   - department;
   - dueAt;
   - createdAt;
   - zoneId;
   - ticketId.
2. Очередь фоновых задач для уведомлений и SLA.
3. Хранилище файлов отдельно от основной БД.
4. Pagination/filters для списков.
5. Серверная аналитика вместо расчета всего на клиенте.
6. Версионирование схемы данных.

## 8. Итоговая оценка

Как демо для руководства система стала значительно убедительнее. Особенно сильны:

- разделение ролей;
- логика уборки;
- автоматическое превращение проблем уборки в жалобы;
- создание ремонтной заявки из поломки;
- видимость для менеджера;
- отчетность и аналитика.

Как production-система она пока не готова. Главная причина не UI, а архитектура: безопасность, база данных, серверные правила, audit trail и надежность хранения.

Правильный следующий шаг: зафиксировать бизнес-процессы в виде деревьев решений, затем по каждому дереву сделать QA matrix: вход, роль, действие, ожидаемый статус, владелец, уведомление, SLA, запись в журнал.
