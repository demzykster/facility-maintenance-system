# Визуальный аудит интерфейса системы

Проверено локально через файл `maintenance-app (1).jsx`.

Локальная версия открывается здесь:

`http://127.0.0.1:5188/`

## Общая оценка

Система выглядит логичной и уже похожа не на простой список заявок, а на рабочий центр управления обслуживанием.

Сильная сторона: интерфейс разделяет роли и процессы. Администратор видит управление, документы, аналитику и настройки. Менеджер и работник не перегружены лишним. Техник видит работу и смену. Уборщик видит свой отдельный контур.

Для презентации руководству это хорошая база, но нужно немного “причесать” демонстрационный слой: добавить примеры заявок, исправить язык в уборке и лучше показать цепочку ответственности.

## Что работает хорошо

### 1. Первый экран входа

Плюсы:

- есть разные входы: команда, работник, техник;
- есть отдельный публичный канал “сообщить без входа”;
- демо-доступ виден сразу, удобно для показа.

Что важно:

- для реального запуска демо-доступ надо убрать или заменить нормальной авторизацией;
- для презентации можно оставить, но сказать: “это демо-режим”.

### 2. Главный экран администратора

Плюсы:

- сразу видны KPI;
- есть очередь документов с истекающими сроками;
- есть уведомления;
- есть быстрый вход в заявки, технику, аналитику, уборку, пользователей, журнал и настройки.

Что стоит улучшить:

- сейчас при нулевых заявках главный экран выглядит как “документы техники”, а не как “центр обслуживания”;
- для презентации надо загрузить 5-8 демонстрационных заявок разных типов.

### 3. Создание заявки

Плюсы:

- сначала простой выбор: здание или транспорт;
- это правильно, потому что пользователь не видит лишнего;
- форма по зданию содержит нужные поля: тема, категория, приоритет, зона, описание, исполнитель, SLA, фото;
- есть AI-анализ по описанию.

Что стоит улучшить:

- для обычного пользователя форма может выглядеть длинной;
- в демо лучше показывать уже заполненный пример;
- кнопку AI стоит объяснять как “помощник классификации”, а не как магию.

### 4. Раздел техники

Плюсы:

- парк техники загружен полно;
- хорошо видны документы, сроки, типы, поставщики;
- фильтры полезные: тип, отдел, поставщик, документы;
- для руководства это сильный раздел, потому что сразу видно контроль активов.

Что стоит улучшить:

- список очень плотный;
- для презентации лучше заранее отфильтровать “проблемные документы” или “просроченные”;
- добавить отдельный вид “топ рисков по технике” на первый экран.

### 5. Аналитика

Плюсы:

- структура правильная: производительность, уборка, обращения работников;
- есть затраты, SLA, среднее время ремонта, повторные зоны, простой, поставщики, нагрузка техников.

Что мешает сейчас:

- без демо-заявок аналитика пустая;
- для презентации обязательно нужны тестовые закрытые и открытые заявки.

### 6. Роли

Проверка показала хорошее разделение:

- работник видит короткий экран “сообщить о проблеме”;
- техник видит очередь, смену и свои задачи;
- уборщик видит только уборочный контур;
- администратор видит всю систему.

Это логически правильно и хорошо объясняется руководству.

## Что нужно срочно исправить

### 1. Ошибка текста в уборке

В разделе уборки слово “зоны” отображается как `זונות`.

На иврите это крайне неудачное и неприемлемое слово. До презентации обязательно заменить.

Рекомендованные варианты:

- `אזורי ניקיון`
- `אזורים`
- `אזורי עבודה`

Например:

- вместо `זונות` использовать `אזורי ניקיון`;
- вместо `אין זונות עדיין` использовать `אין אזורי ניקיון עדיין`;
- вместо `לא הוגדרו זונות` использовать `לא הוגדרו אזורי ניקיון`.

### 2. Добавить демо-сценарии

Сейчас открытых заявок ноль, поэтому руководству будет сложно увидеть процесс.

Нужны демо-данные:

- транспортная заявка “техника не передана технику”;
- заявка здания “в работе”;
- заявка от работника “ожидает менеджера”;
- заявка “ожидает подтверждения результата”;
- заявка “ожидает финального закрытия”;
- закрытая заявка со стоимостью;
- просроченная SLA;
- жалоба по уборке;
- пропущенное окно уборки.

### 3. Сильнее показать “у кого сейчас мяч”

Логика в системе есть, но визуально ее стоит сделать главным управленческим элементом.

В каждой заявке желательно показать:

- следующий ответственный;
- почему именно он;
- сколько времени заявка находится у него;
- что нужно сделать следующим действием.

Пример:

`Следующий шаг: менеджер должен подтвердить результат`

или:

`Следующий шаг: менеджер должен передать технику технику`

## Что стоит дополнить

### 1. Презентационный режим

Сделать кнопку или набор данных “Демо для руководства”.

Она должна загружать:

- реальные на вид заявки;
- разные статусы;
- историю действий;
- стоимость;
- фото-заглушки;
- уборку;
- просрочки;
- аналитику.

### 2. Главная очередь решений

На дашборде стоит добавить блок:

- требует администратора;
- требует менеджера;
- требует техника;
- ожидает внешнего фактора;
- просрочено SLA.

Это лучше всего объясняет управленческую ценность системы.

### 3. Карта проблем

Для руководства полезно:

- проблемные зоны здания;
- проблемная техника;
- частые причины простоев;
- частые причины ожидания;
- топ поставщиков по затратам;
- уборочные зоны с жалобами.

### 4. Отдельный отчет “почему простаивает техника”

Очень сильный управленческий отчет:

- ремонт;
- не передали технику технику;
- нет запчастей;
- поставщик;
- нет доступа;
- ожидание решения менеджера.

## Итог

Логика системы правильная. Архитектура ролей и статусов выглядит зрелой.

Главное перед презентацией:

1. Исправить тексты уборки.
2. Загрузить демонстрационные заявки.
3. Сделать акцент на “кто сейчас должен действовать”.
4. Показывать не пустую аналитику, а подготовленный сценарий.
5. Вынести управленческие KPI в первый экран.
