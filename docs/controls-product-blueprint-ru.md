# Концепция Модуля `בקרות`

Рабочий продуктовый документ по тому, что мы хотим создать.

Статус: согласованный черновик по итогам диалога 2026-07-03.  
Назначение: сохранить направление, чтобы следующая сессия/разработчик/Клод не восстанавливали логику по памяти.

Этот документ не является аудитом текущих багов. Он описывает целевый продукт: как должен работать будущий модуль `בקרות`, какие процессы он должен покрывать, какие роли участвуют, как рождаются проверки, замечания, задачи, отчеты и управленческие выводы.

## 1. Главная Идея

`בקרות` должен стать не отдельной страницей "инспекции", а общим движком контроля процессов компании.

Это единая система для:

- проверок безопасности (`בטיחות`);
- контроля качества (`בקרת איכות`);
- проверки транспорта и рабочих средств (`בקרת כלים`);
- обходов территорий, офисов, столовой, склада, наружных зон;
- обходов руководства (`סיור הנהלה`, `סיור מנכ"ל`);
- фиксации замечаний, отклонений, рисков, улучшений и повторяющихся проблем;
- назначения follow-up действий через задачи, тикеты, уведомления или отчетность;
- управленческого dashboard с трендами, критичными отклонениями и просрочками.

Ключевой принцип: один общий control engine, внутри которого есть разные домены, а не отдельные несвязанные модули для safety, quality, fleet и management walks.

## 2. Почему Не Делаем Узкую "Инспекцию"

Текущая мысль шире обычного checklist-модуля.

В компании есть разные типы контроля:

- ממונה בטיחות проверяет безопасность;
- менеджеры и руководители делают обходы своих зон;
- рабочие могут проверять свой транспорт или рабочее место;
- QA проверяет процессы TPL-логистики;
- руководство смотрит на общую картину, повторы, тренды и критичные отклонения;
- ועדת בטיחות / צוות חירום / נאמני בטיחות участвуют в дополнительных проверках и обсуждениях помимо основной должности.

Поэтому система должна уметь обслужить не только "прошел/не прошел чеклист", а полный цикл:

```text
Программа контроля
  -> планирование / назначение
  -> выполнение обхода
  -> findings / замечания
  -> решение что делать
  -> задача / тикет / отчет / уведомление
  -> follow-up / закрытие
  -> аналитика / повторяемость / dashboard
```

## 3. Домены Внутри `בקרות`

### 3.1 Safety / `בטיחות`

Покрывает:

- обходы зон и участков;
- нарушения PPE;
- опасные действия;
- проблемы инфраструктуры, создающие риск;
- повторяющиеся нарушения;
- участие ועדת בטיחות, נאמני בטיחות, צוות חירום;
- подготовку к аудитам и визитам проверяющих;
- safety meeting / forum context.

Нарушение в safety не означает, что вся проверка провалена. Проверка может быть выполнена, но внутри нее есть одно или несколько замечаний.

### 3.2 Quality / `בקרת איכות`

Это не маленькое дополнение к safety, а отдельный крупный домен внутри того же движка.

Контекст компании: TPL-логистика. Контроль качества должен учитывать разные процессы:

- сбор заказов на палеты;
- сбор заказов в пластиковые контейнеры;
- обработка `החזרות`;
- `פגומים`;
- поврежденная или проблемная `סחורה`;
- разные категории товаров;
- разные способы работы по клиентам и процессам;
- проверки проблемных работников или проблемных участков;
- проверки конкретных заказов, партий, процессов, смен или работников;
- возможность добавлять клиентов в будущем, если появятся разные клиентские требования.

QA-домен должен поддерживать гибкость:

- регулярные проверки процесса;
- выборочную проверку заказов;
- проверки по триггеру: жалоба, ошибка, подозрение, проблемный работник, проблемная зона;
- sampling policy: сколько проверять, как часто, по каким правилам;
- scoring/rating, если это настроено;
- чувствительную видимость по работникам.

### 3.3 Fleet / Tools / `בקרת כלים`

Транспортные проверки не должны оставаться отдельной параллельной вселенной.

`בקרת כלים` становится одним доменом внутри `בקרות`.

Примеры:

- проверка погрузчика;
- проверка документации/состояния;
- проверка перед работой;
- периодические проверки;
- проверка конкретного типа техники;
- finding по технике может открыть maintenance ticket или task.

### 3.4 Operations / Area Walks

Покрывает:

- складские зоны;
- наружную территорию;
- офисы;
- столовую;
- приемку;
- возвраты;
- зоны отгрузки;
- упаковку;
- общие операционные обходы.

### 3.5 Management / CEO Walks

Например `סיור מנכ"ל`.

Это гибридный тип контроля: не только safety и не только quality. Руководители могут проходить по складу и фиксировать:

- safety issues;
- quality issues;
- logistics/process issues;
- improvement ideas;
- responsible person/department;
- several responsible people;
- category;
- due date;
- status.

Сейчас такие обходы могут вестись вручную секретарем. В целевой системе это становится структурированным control run, где участники, замечания и действия сохраняются в общей модели.

## 4. Люди, Роли И Группы

Важно разделять:

- должность/роль в системе;
- права доступа;
- членство в группе;
- участие в конкретной программе контроля;
- ответственность за конкретное действие.

Один человек может быть:

- руководителем департамента;
- членом ועדת בטיחות;
- участником צוות חירום;
- נאמן בטיחות;
- руководителем одной из групп;
- наблюдателем процессов;
- ответственным за конкретное finding/action.

### 4.1 User Groups

Нужна модель `userGroups`.

Примеры групп:

- `ועדת בטיחות`;
- `צוות חירום`;
- `נאמני בטיחות`;
- QA team;
- Operations leadership;
- Department managers;
- Executive observers.

Группа не равна роли. Это дополнительный слой над пользователями.

Пользователь может быть в нескольких группах. Некоторые группы могут иметь руководителя. Некоторые группы могут получать уведомления, видеть отчеты или участвовать в назначениях.

### 4.1.1 Cleaning Workers

Согласованное направление: `cleaner` не должен оставаться долгосрочной системной ролью.

Уборщик — это обычный `worker`, у которого есть постоянный доступ/группа уборки.

Целевая логика:

- `role: "worker"`;
- worker-number/PIN login как у остальных работников;
- worker-функции остаются доступны;
- уборка задается через cleaning access/capability/group;
- подрядчик задается через `employmentType: "contractor"` и `contractorName`, а не отдельной ролью.

Обычный worker с уборкой может:

- выполнять обходы уборки;
- получать жалобы по своим/покрываемым зонам;
- закрывать/отвечать на жалобы.

Управление зонами, окнами, чеклистами и аналитикой уборки остается у пользователей с management/module permissions.

Переход должен быть совместимым: старый `role === "cleaner"` работает до тех пор, пока код, серверные permission checks и login/session paths не переведены на helpers.

### 4.1.2 User Activity Status

Статус входа не должен перегружать форму создания нового пользователя. При создании достаточно собрать профильные поля и доступы; first-login/secret setup уже описан onboarding flow.

Для уже существующего профиля позже нужен компактный статус активности:

- небольшой индикатор рядом с именем или под именем;
- активен/заблокирован/ожидает first login;
- последний визит, если он известен;
- при необходимости - ссылка на подробную историю входов.

Это нельзя строить из локального UI-состояния или "примерного" времени. Нужны реальные серверные данные: `lastSeenAt`, append-only `loginEvents`/session events и audit trail. В будущем эти данные могут стать частью аналитики активности пользователей и управленческого dashboard.

### 4.2 Кто Выполняет Проверки

Выполнять могут:

- ממונה בטיחות;
- менеджеры;
- руководители;
- QA сотрудники;
- назначенные участники userGroups;
- рабочие, если программа требует self-check своего транспорта/рабочего места.

Назначение должно быть гибким не только по роли, а по конкретным персонам и группам.

## 5. Назначения И Расписание

Цель не просто создать список проверок, а обеспечить нужный coverage без перегруза и дублей.

### 5.1 Программы Контроля

Программа контроля определяет:

- что проверять;
- кого или что проверять;
- кто может выполнять;
- как часто;
- нужен ли checklist;
- нужны ли фото;
- что делать с findings;
- кто видит результат;
- какие SLA;
- какие уведомления;
- как распределять нагрузку.

### 5.2 Удобные Дни Исполнителей

У конкретного человека могут быть удобные дни недели.

Пример: менеджеру удобнее делать обходы в воскресенье и четверг.

Система должна учитывать:

- preferred weekdays;
- выходные;
- праздники;
- корпоративные нерабочие дни;
- отсутствие/перегруз сотрудника;
- максимальную допустимую задержку.

Перед тем как фиксировать схему расписания, нужно вручную прогнать один реальный сценарий:

```text
Менеджер отдела
  удобные дни: воскресенье и четверг
  target: конкретная зона
  anti-duplicate window: 30 дней
  coverage threshold: минимум 1 раз в месяц
  max delay: по настройке программы
```

Первый PR не обязан строить полный scheduling engine, но поля `Program` и `Assignment` должны доказать, что такой сценарий описывается без костылей.

### 5.3 Нерабочие Дни

Должна быть общая настройка нерабочих дней:

- праздники;
- корпоративные дни;
- закрытые дни;
- специальные исключения.

Если на такой день уже назначен обход, система должна сдвигать или предлагать перерасчет.

Право управлять такими днями должно быть настраиваемым. Это не только admin.

### 5.4 Переносы И Переназначения

Пользователь должен иметь удобный способ сказать:

- я отсутствую;
- я загружен;
- перенести на другой допустимый день;
- переназначить другому человеку, если есть право.

Система не должна давать произвольный календарь без правил. Она должна учитывать:

- крайний срок;
- допустимую задержку;
- загруженность;
- дубли;
- coverage threshold;
- права пользователя.

### 5.5 Защита От Дублей И Недостаточного Coverage

Нужно предотвращать ситуации:

- один и тот же объект проверяют несколько раз за короткий период без причины;
- зона вообще не проверяется месяцами;
- один человек получает слишком много обходов;
- система формально назначает проверки, но не достигает минимального месячного/годового покрытия.

Важен не только вопрос "кто делает", а вопрос "выполняем ли мы минимальный порог контроля за период".

## 6. Обход / Run

Обход - это фактическое выполнение назначенной проверки.

В нем должны быть:

- программа;
- исполнитель;
- участники;
- дата/время;
- объект/зона/процесс;
- checklist;
- answers;
- findings;
- общий комментарий;
- подпись за весь обход.

Подпись нужна не за каждый пункт checklist, а за обход в целом.

Checklist item может быть:

- OK;
- не OK;
- не релевантно;
- замечание;
- требует фото при проблеме;
- требует follow-up.

## 7. Findings / Замечания

Finding - это отдельная запись о проблеме, отклонении, риске или наблюдении.

Важно: finding не означает, что вся проверка "провалена". Проверка может быть выполнена успешно, но иметь одно или несколько замечаний.

Finding должен хранить:

- домен;
- программу;
- run;
- location;
- объект проверки;
- checklist item;
- описание;
- severity;
- category;
- фото;
- кого касается;
- кто создал;
- кому видно;
- какие действия приняты;
- связь с задачей/тикетом/report-only.

### 7.1 Если Не Известно Имя Работника

Система должна поддерживать неполное знание.

Например:

1. Это наш работник или подрядчик?
2. Если подрядчик - какой подрядчик, если известно?
3. Если наш работник - чей отдел?
4. Известен ли руководитель?
5. Известно ли имя работника?

Не всегда инспектор знает имя человека. Но он может знать зону, отдел, подрядчика или руководителя. Это должно быть достаточно для полезной записи.

### 7.2 Повторяемость

Система должна видеть повторяющиеся проблемы:

- одна и та же зона;
- один и тот же процесс;
- один и тот же тип нарушения;
- один и тот же работник/команда/подрядчик;
- одно и то же оборудование;
- повтор после предыдущего решения.

Нужно показывать не только факт повтора, но и какие решения принимались раньше.

## 8. Что Делать С Finding

После finding система должна помочь выбрать действие.

Варианты:

- только для отчета;
- создать задачу в `מטלות`;
- открыть maintenance ticket / `קריאה`;
- назначить follow-up;
- уведомить руководителя;
- уведомить группу;
- передать в QA;
- передать в maintenance;
- training / беседа с работником;
- CAPA-lite;
- supplier/customer related action.

Например:

- сломанное оборудование или угроза безопасности -> maintenance ticket;
- работник без каски -> задача руководителю провести беседу;
- легкое наблюдение -> report-only;
- повторяющаяся QA ошибка -> задача QA/руководителю процесса;
- проблема с товаром/заказом -> quality action, possibly linked to order/customer/process.

## 9. `מטלות` Как Общий Action Layer

`מטלות` должны стать общим слоем follow-up действий.

Это не только "задачи обслуживания". Через них могут закрываться:

- findings из safety;
- findings из quality;
- management walk actions;
- cleaning follow-ups;
- ticket-related tasks;
- training/conversation tasks;
- operational improvements.

Будущая задача должна знать источник:

- `sourceType`;
- `sourceId`;
- `findingId`;
- `locationId`;
- `categoryId`;
- `severity`;
- `assignedTo`;
- `dueDate`;
- `status`;
- `closureEvidence`.

Но при этом пользователь видит один понятный список задач, а не десять разных потоков.

## 9.1 `פגישות` / `פ.ע` Как Рабочее Событие

Встречи не должны превращаться в отдельную мини-систему, которая живет отдельно от `בקרות`, `מטלות` и `קריאות`.

При этом обычные `פ.ע` остаются важным самостоятельным сценарием. Их нельзя потерять ради `בקרות`.

Базовая модель:

- одна сущность `פגישה`;
- обычная `פ.ע` / regular meeting - первый и самый простой контекст;
- встреча может быть создана вручную или из другого источника;
- "с менеджером", "с коллегой", "групповая" лучше определять по участникам, а не заставлять пользователя выбирать как обязательный тип.

Вместо текущего жесткого `סוג פגישה` лучше развивать мягкое поле `meetingContext` / `הקשר פגישה`:

- `רגילה / פ.ע`;
- `ישיבת צוות`;
- `ועדה / פורום`;
- `בעקבות בקרה`;
- `בעקבות מטלה`;
- `בעקבות קריאה`;
- `סיור הנהלה`;
- `אחר`.

Если встреча создается из `בקרות`, `מטלה` или `קריאה`, контекст и source link должны заполняться автоматически.

Будущая meeting record должна поддерживать:

- `sourceModule`;
- `sourceId`;
- `sourceLabel`;
- `meetingContext`;
- `participantIds`;
- agenda / `סדר יום`;
- decisions / `החלטות`;
- action items / linked `מטלות`;
- linked findings / runs / tickets where relevant.

Цель: встреча становится частью цепочки управления:

```text
event/finding/task/ticket
  -> discussion / פגישה
  -> decision
  -> action item / מטלה / קריאה
  -> follow-up / closure
```

На первом UI шаге не нужно усложнять форму. Можно убрать обязательный выбор "тип встречи" и оставить обычную `פ.ע` как default, показывая `הקשר` только если встреча пришла из другого объекта или если пользователь открыл advanced options.

## 10. `קריאות` / Maintenance Tickets

Не каждое finding должно становиться ticket.

Ticket нужен, когда есть:

- поломка;
- инфраструктурная проблема;
- угроза безопасности, которую должен решать maintenance;
- техническое действие;
- ремонт;
- блокировка/восстановление оборудования.

Нужно не плодить второй поток для maintenance. Если проблема относится к maintenance, она должна идти в существующий ticket flow, но с нормальной ссылкой на source finding.

## 11. Locations / Zones

Нужна общая база зон и мест.

Не должно быть трех разных списков:

- cleaning zones;
- maintenance zones;
- controls zones.

Целевое направление:

- общий объектный справочник `locations`;
- UI может показывать его в разных местах;
- cleaning/settings/controls используют одну базу;
- добавление/удаление/переименование должно обновлять связанные места;
- права на управление location должны быть настраиваемыми.

Примеры location:

- складская зона;
- ramp;
- receiving;
- returns;
- damaged goods area;
- office;
- canteen;
- outdoor area;
- parking;
- maintenance room;
- charging area;
- general.

## 12. Видимость, Права И Чувствительность

Нужны гибкие настройки видимости.

Некоторые записи можно показывать широко, некоторые - только ограниченному кругу.

Пример sensitive quality/safety case:

- "проблемный работник" видят только QA/руководитель/операционный директор;
- широкая группа видит агрегат или тренд без имени;
- руководитель департамента видит действия по своему отделу;
- ועדת בטיחות видит safety context;
- руководство видит trend/critical status, но не каждую мелкую строку.

Права должны быть:

- на доступ к модулю;
- на управление программами;
- на выполнение проверок;
- на просмотр findings;
- на просмотр sensitive findings;
- на назначение действий;
- на закрытие действий;
- на управление календарем нерабочих дней;
- на управление userGroups (`userGroups:view/manage` как отдельный module access, не часть общей роли);
- на управление locations.

Модуль `בקרות` должен быть добавлен в настройки module access.

## 13. Dashboard Для Руководства

Для סמנכ"ל תפעול, מנכ"ל и руководства основной инструмент - не список всех records, а умный dashboard.

Он должен показывать:

- критичные отклонения;
- повторяемость;
- тренды;
- просрочки;
- unresolved high severity;
- зоны с частыми проблемами;
- процессы с падением качества;
- задачи без закрытия;
- данные из `קריאות`, `ניקיון`, `מטלות`, `בקרות`.

Идея: руководство имеет доступ не столько к каждому модулю, сколько к smart BI / attention layer.

Dashboard не должен заваливать мелким шумом. Он должен отвечать: "на что нужно обратить внимание".

Чтобы такой dashboard потом реально построить, у разных сущностей должен быть минимальный общий "сигнальный конверт":

```js
{
  severity,
  status,
  assignedTo,
  dueAt,
  sourceModule,
  sourceId
}
```

Это не значит, что все модули должны храниться одинаково. Это значит, что finding, task, ticket, cleaning signal и SLA alert должны быть читаемы как сравнимые сигналы, а не требовать отдельный ручной парсер для каждого источника.

## 14. QA Responsibility Triangle

Для quality логика строится вокруг треугольника:

```text
QA / Quality Department
  <-> Department Managers
  <-> Process Owners / Workers / Contractors
```

QA может:

- проводить проверки;
- находить отклонения;
- связывать их с процессом/заказом/работником/зоной;
- отправлять follow-up руководителю;
- отслеживать повторяемость;
- показывать trend руководству.

Руководитель департамента:

- получает relevant findings;
- отвечает за исправление;
- может назначать подзадачи;
- видит sensitive данные по своему периметру.

Операционный директор / руководство:

- видит общую картину;
- видит тенденции;
- видит критичные точки;
- не получает поток мелких записей без фильтра.

## 15. AI В Будущей Системе

AI должен быть не декоративным чатиком, а помощником внутри правильной модели.

AI может помогать:

- классифицировать finding;
- предложить severity;
- предложить route: ticket / task / report-only;
- найти похожие повторяющиеся проблемы;
- предложить responsible group/person;
- подготовить summary для dashboard;
- объяснить, почему проблема стала повторяющейся;
- помогать QA/safety смотреть на trend.

Но AI не должен напрямую писать в данные.

Любое действие должно проходить через:

- права пользователя;
- product operation;
- validation;
- audit trail;
- подтверждение, если действие значимое.

AI должен быть server-side / edge-side, не browser API call с provider key.

## 15.1 Настройки И Владение Модулями

Перед `בקרות` важно не просто "почистить визуал" общих настроек, а разложить настройки по смыслу.

Согласованная линия:

- `ברירת מחדל לתדירות בקרה` устарела как пользовательская настройка. В новой логике проверки задаются программами: fleet inspection program, controls program, QA program. Глобальная частота будет путать. Ее можно убрать из UI, оставив технический fallback в коде, если он еще нужен старым участкам.
- `קיבולת טיפולים יומית` живая настройка, но это не общая настройка компании. Она относится к периодическим ТО и должна жить рядом с `לוח טיפולים`, генерацией по регуляциям и PM rules. Переносить ее стоит только когда у `כלי שינוע` уже есть понятный settings/home для PM.
- `תזכורת לפני סבב ניקיון` живая настройка уборки. Ее место в `בקרת ניקיון`, рядом с окнами/rounds. Но переносить ее сейчас без стабильного cleaning settings surface не обязательно: лучше сначала создать правильный дом, потом переносить.
- `סוגי התראות` это не личные уведомления пользователя, а системная политика уведомлений. Если оставить в общих настройках временно, блок должен быть компактным/advanced и ясно называться `מדיניות התראות מערכת`. Личные фильтры остаются в панели уведомлений.

Главный принцип: не переносить живые настройки просто ради красоты. Сначала должен появиться правильный модульный дом и понятный save handler. Устаревшие настройки можно убрать раньше, потому что они создают ложную модель.

## 16. Первый Вертикальный Срез

Первый implementation slice должен быть маленьким, но настоящим.

Предлагаемый минимальный сценарий:

1. Создать control program.
2. Выбрать domain: safety / quality / fleet.
3. Настроить checklist.
4. Назначить конкретному человеку или группе.
5. Выполнить run.
6. Создать finding.
7. Выбрать action:
   - report-only;
   - task;
   - ticket.
8. Увидеть это в списке задач/dashboard.

Важно: с первого этапа транспорт (`בקרת כלים`) должен быть одним из use cases внутри `בקרות`, а не отдельным будущим переносом.

Первый quality slice должен быть намеренно узким:

- один QA-процесс;
- один тип finding;
- один action route;
- без worker scoring;
- без CAPA;
- без customer SLA;
- без широкой sampling automation;
- без полноценного quality BI.

## 17. Что Не Делаем Сразу

Не делаем сразу:

- полный monolith split;
- полную нормализацию всех таблиц;
- автоматический AI;
- сложную миграцию старых inspection data;
- отдельный новый поток maintenance work вместо существующих `קריאות`;
- десять dashboard сразу;
- идеальную scheduling engine в первом PR.
- полноценную систему `בקרת איכות` в первом slice.

Сначала foundation и вертикальный срез.

## 18. Принцип Реализации

Работа должна идти маленькими PR:

1. Уже сделано: зафиксировать документы и текущий аудит.
2. Уже сделано: убрать legacy inspection/questionnaire хвосты.
3. Уже сделано: подготовить `מטלות` как action layer.
4. Уже сделано: подготовить общий location model на уровне helpers/tests.
5. Ближайшее: подготовить `cleaningAccessModel`, чтобы `cleaner` стал legacy-compatible переходом к `worker + cleaning access`.
6. Затем: заменить прямые проверки `role === "cleaner"` на helpers и обновить server/KV/session policies.
7. Затем: подготовить `userGroups`.
8. Затем: создать core models для `בקרות`.
9. Затем: skeleton `בקרות` и первый vertical slice.
10. Затем: scheduling, domains, dashboard/attention layer, server-side AI assistance.

Главный принцип: не строить новую большую систему поверх старой путаницы. Сначала убрать противоречия, затем добавить общий движок, затем расширять домены.

## 19. Короткое Резюме

Мы хотим создать модуль `בקרות` как общий управленческий и операционный слой контроля.

Он должен объединить safety, quality, fleet/tools, operations и management walks вокруг одной модели:

```text
Program -> Assignment -> Run -> Finding -> Action -> Follow-up -> Insight
```

Важнейшие решения:

- не плодить отдельные inspection-системы;
- не тащить legacy questionnaires;
- не делать `בקרת כלים` отдельным миром;
- использовать `מטלות` как общий action layer;
- locations сделать общей базой;
- cleaning workers должны быть `worker + cleaning access`, а `cleaner` остается только legacy bridge на переходе;
- userGroups сделать отдельным слоем над ролями;
- dashboard для руководства должен показывать важное, а не всё подряд;
- AI добавлять позже и только поверх правильной server-side модели.
